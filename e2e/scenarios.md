# Интеграционные сценарии

Этот документ фиксирует, какие пользовательские сценарии покрыты автоматическими e2e-тестами в `e2e/tests/`. Соответствует требованию шага 4: «Опишите и зафиксируйте основные пользовательские сценарии для проверки».

Тесты гоняют **реальный браузер** (Chromium через Playwright) против **реального бэкенда** (Fastify, in-memory) и **реального фронтенда** (Vite preview), без моков. Бэкенд стартует с фиксированным clock (`FIXED_CLOCK_ISO=2026-06-01T09:00:00Z`, понедельник 12:00 MSK) — это нужно для детерминированной 14-дневной сетки слотов.

## Сценарий 1 — Бронирование (happy path)

**Цель:** убедиться, что главный путь гостя проходит сквозь UI → API → доменную логику → подтверждение, и что слот, забронированный через UI, на бэке действительно становится `booked` (виден в сетке как занятой).

**Покрывает:**
- `GET /event-types` — список типов событий из seed.
- `GET /slots?eventTypeId=…` — 14-дневная сетка слотов с правильными статусами.
- `POST /bookings` — создание бронирования с возвратом `GuestBookingConfirmation` (title, длительность, времена) без второго запроса.
- инвариант «один слот = одна бронь»: после бронирования тот же слот в сетке имеет статус `booked`.

**Шаги:**

| # | Действие | Ожидание |
|---|---|---|
| 1 | Перейти на `/` | На странице видна карточка типа «Вводный звонок» (из seed). |
| 2 | Клик по карточке «Вводный звонок» | URL = `/event-types/intro-call`, отрисована сетка слотов. |
| 3 | Найти первый available-слот (по `aria-label` с суффиксом `(available)`) и запомнить его подпись. Клик по нему. | URL = `/event-types/intro-call/book?slot=…`, отрисована форма. |
| 4 | Заполнить `#guestName` = `Иван Тестов`, `#guestEmail` = `ivan@test.io`, нажать «Забронировать». | Перезод на `/bookings/success`. |
| 5 | На экране успеха проверить заголовок «Встреча забронирована!», название типа «Вводный звонок», имя и email гостя. | Все элементы видны. |
| 6 | Кликнуть «Вернуться к типам встреч», вернуться в сетку слотов того же типа, найти тот же самый слот по запомненной подписи. | У слота `aria-label` теперь имеет суффикс `(booked)` — занят. |

## Что НАМЕРЕННО не покрывается e2e

Эти инварианты уже покрыты юнитами/контрактом и в e2e не дублируются:

- Валидация полей формы (zod на фронте + Ajv на бэке) — `frontend/src/routes/BookingFormPage.tsx`, тесты бэка.
- 409 при двойном бронировании на уровне API — `backend/test/bookings.test.ts`.
- 14-дневное окно и пересечение интервалов — `backend/test/slots.test.ts`, `backend/test/bookings.test.ts`.
- Админка (создание `EventType`, список `AdminBooking`) — покрыта контрактом и юнитами; в e2e — только если расширим сценарии.

## Запуск

```sh
cd e2e && npm ci && npx playwright install chromium
cd e2e && npm test
```

`webServer` в [playwright.config.ts](playwright.config.ts) сам поднимает backend (`:3000`, с фиксированным clock) и frontend preview (`:4173`). Зависимости — backend и frontend должны быть собраны (`npm run build`) до запуска тестов.
