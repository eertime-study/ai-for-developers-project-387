# Backend — Календарь звонков

Серверная часть (шаг 3). Реализует API **строго по контракту** `spec/main.tsp` →
`openapi.yaml`. Хранение — in-memory.

## Стек

- **Node.js + TypeScript** (ESM, `tsconfig` NodeNext + strict)
- **Fastify** — HTTP-сервер
- **fastify-openapi-glue** — роуты и валидация запросов прямо из `openapi.yaml` (operationId)
- **Ajv + ajv-formats** — валидация форматов контракта (`date-time`, `int32`, `email`)
- **openapi-typescript** — генерация типов из контракта (`src/api/schema.ts`)
- **vitest** + `app.inject()` — тесты

## Запуск

Предусловие: контракт должен быть скомпилирован (тот же `openapi.yaml`, что у фронта).

```sh
# 1. из корня репозитория — собрать контракт
cd spec && npm install && npm run compile

# 2. backend
cd backend && npm install
npm run gen:api      # типы из openapi.yaml -> src/api/schema.ts
npm run dev          # tsx watch, сервер на :3000
# или прод-режим:
npm run build && npm start
```

Порт настраивается через `PORT` (по умолчанию `3000`), хост — `HOST` (по умолчанию `0.0.0.0`).

CORS разрешён для `http://localhost:5173` (Vite dev) и `http://localhost:4173` (`vite preview`).

## Тесты

```sh
npm test
```

Каждый тест поднимает свежее приложение (свежий store) с фиксированным `clock`; покрыты
успешные и ошибочные сценарии (двойное бронирование, пересечение интервалов, нормализация
offset, невалидный email, прошлое/вне окна, дубликат типа события, формат ошибок).

## Эндпоинты (по контракту)

| Метод | Путь | Описание |
|---|---|---|
| GET | `/owner` | профиль владельца календаря |
| GET | `/event-types` | список типов событий |
| GET | `/event-types/{id}` | один тип события (404, если нет) |
| GET | `/event-types/{id}/slots` | сетка слотов на 14 дней |
| POST | `/bookings` | бронирование слота (400/404/409) |
| POST | `/admin/event-types` | создание типа события (400/409) |
| GET | `/admin/bookings` | предстоящие встречи в обогащённом виде |

Все успешные ответы — статус **200** (как в контракте, в т.ч. POST). Тело ошибок —
`{ code, message, field? }` с кодом из `ErrorCode` контракта.

## Важные оговорки (caveats)

- **Данные in-memory** и сбрасываются при каждом рестарте — отдельная БД на этом шаге не нужна.
- **Сетка слотов строится в UTC.** Человеческое представление времени формирует фронт по
  `owner.timeZone`.
- **Рабочие часы не реализуются** — такого правила нет в контракте/задании; `outside_window`
  означает только «вне 14-дневного окна».
- **`openapi.yaml` (из `spec/main.tsp`) — source of truth.** Контракт в рамках шага 3 **не менялся**.

## Бизнес-правила (на бэкенде)

- **Анти-двойное бронирование по пересечению интервалов**, независимо от типа события:
  новый интервал `[start, end)`, пересекающийся с существующей бронью, → `409 SLOT_ALREADY_BOOKED`.
- `startTime` нормализуется (`toISOString`) перед проверкой и записью — один и тот же момент в
  разных offset нельзя забронировать дважды.
- Окно записи — ровно 14 дней от текущего момента; вне окна → `SLOT_OUTSIDE_BOOKING_WINDOW`,
  в прошлом → `SLOT_IN_PAST`.
