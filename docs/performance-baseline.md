# Performance baseline (Lighthouse)

Базлайн производительности для регулярного аудита из
[.github/workflows/lighthouse.yml](../.github/workflows/lighthouse.yml).

Файл ведёт **OpenCode-агент**: на каждом ночном (или ручном) прогоне он сравнивает свежие
показатели Lighthouse с таблицами ниже, помечает регрессии >10% в issue `[lighthouse]`, а затем
перезаписывает таблицы новыми значениями. Так фиксируется тренд между прогонами.

- Категории — оценка Lighthouse 0–100 (выше лучше).
- Lab-метрики — из лабораторного прогона Lighthouse (не field): **LCP**, **TBT**, **CLS**, **FCP**,
  **Speed Index**. FID/INP здесь не приводятся — это field-метрики, в lab Lighthouse их нет; за
  интерактивность в lab отвечает TBT.
- Источник чисел — медиана из `numberOfRuns` представительного прогона (`isRepresentativeRun`).

> Последний прогон: **2026-06-17 (UTC, ночной)**. Категории (100/100/96/90), TBT и CLS без
> изменений на обеих страницах. **Есть регрессии >10%** (абсолютные значения остаются хорошими,
> Performance = 100). Главная: LCP 479 → 540 ms (**+12.7%, регрессия**), FCP 473 → 495 ms (+4.7%),
> Speed Index 598 → 496 ms (−17.1%, улучшение). Админка: LCP 482 → 541 ms (**+12.2%, регрессия**),
> FCP 482 → 541 ms (**+12.2%, регрессия**), Speed Index 482 → 754 ms (**+56.4%, регрессия**).
> Вероятная причина — холодный старт Render free tier + сетевая латентность до API (контент
> блокируется на API-запросах, SPA рендерит в пустой `#root`); Speed Index админки скачет из
> прогона в прогон. Стоячие потолки категорий: SEO 90 (нет `<meta name="description">`),
> Best Practices 96. Детали и рекомендации в issue `[lighthouse]`. Следующий прогон сравнивается
> с самой нижней строкой каждой таблицы.
