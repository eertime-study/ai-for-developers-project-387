# Performance baseline (Lighthouse)

Базлайн производительности для регулярного аудита из
[.github/workflows/lighthouse.yml](../.github/workflows/lighthouse.yml).

Файл ведёт **OpenCode-агент**: на каждом ночном (или ручном) прогоне он сравнивает свежие
показатели Lighthouse с таблицами ниже, помечает регрессии >10% в issue `[lighthouse]`, а затем
перезаписывает таблицы новыми значениями. Так фиксируется тренд между прогонами.

- Категории — оценка Lighthouse 0–100 (выше лучше).
- Lab-метрики — из лабораторного прогона Lighthouse (не field): **LCP**, **TBT**, **CLS**, **FCP**,
  **Speed Index**. FID/INP здесь не приводятся — это field-метрики, в lab Lighthouse их нет; за
  интерактивность в lab отвечает TBT.
- Источник чисел — медиана из `numberOfRuns` представительного прогона (`isRepresentativeRun`).

> Последний прогон: **2026-06-19 (UTC, ночной)**. Категории почти стабильны: админка
> 100/100/96/90; главная просела Performance 100 → 99 (−1 пункт, округление шумной метрики).
> TBT и CLS без изменений на обеих страницах. **Зафиксированы регрессии >10% в lab-метриках
> загрузки** (формально по правилу аудита): главная — LCP 533 → 680 ms (+27.6%),
> FCP 487 → 674 ms (+38.4%), Speed Index 507 → 1074 ms (+111.8%); админка — LCP/FCP 480 → 543 ms
> (+13.1%), Speed Index 480 → 858 ms (+78.75%). Диагноз прежний: колебания — холодный старт
> Render free tier + сетевая латентность до API (контент блокируется на API-запросах, SPA рендерит
> в пустой `#root`), а не деградация кода (код фронта между прогонами не менялся); FCP=LCP это
> подтверждают, Speed Index — самый «прыгучий». Стоячие потолки категорий: SEO 90 (нет
> `<meta name="description">`), Best Practices 96. Детали и рекомендации в issue `[lighthouse]`.
> Следующий прогон сравнивается с самой нижней строкой каждой таблицы.
