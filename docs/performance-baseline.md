# Performance baseline (Lighthouse)

Базлайн производительности для регулярного аудита из
[.github/workflows/lighthouse.yml](../.github/workflows/lighthouse.yml).

Файл ведёт **OpenCode-агент**: на каждом ночном (или ручном) прогоне он сравнивает свежие
показатели Lighthouse с таблицами ниже, помечает регрессии >10% в issue `[lighthouse]`, а затем
перезаписывает таблицы новыми значениями. Так фиксируется тренд между прогонами.

- Категории — оценка Lighthouse 0–100 (выше лучше).
- Lab-метрики — из лабораторного прогона Lighthouse (не field): **LCP**, **TBT**, **CLS**, **FCP**,
  **Speed Index**. FID/INP здесь не приводятся — это field-метрики, в lab Lighthouse их нет; за
  интерактивность в lab отвечает TBT.
- Источник чисел — медиана из `numberOfRuns` представительного прогона (`isRepresentativeRun`).

> Последний прогон: **2026-07-04 (UTC, ночной)**. Категории стабильны: обе страницы
> 100/100/96/90. TBT и CLS без изменений. **Регрессий >10% нет.**
> **Главная:** LCP 517 → 532 ms (+2.9%, шум), FCP 477 → 485 ms (+1.7%, шум),
> Speed Index 510 → 485 ms (−4.9%, улучшение). **Админка:** LCP/FCP 483 → 477 ms (−1.2%, шум),
> Speed Index 579 → 625 ms (+7.9%, рост в пределах коридора, ниже порога).
> Разброс LCP≈FCP при неизменных Performance=100, TBT=0 — вариативность Render free tier
> (холодный старт / отдача статики), а не деградация кода: SPA рендерит в пустой `#root`. Значения
> внутри исторического коридора таблиц (главная FCP 468–674 ms, LCP 479–680 ms; админка 467–866 ms).
> Стоячие потолки категорий: SEO 90 (нет `<meta name="description">` в `frontend/index.html`),
> Best Practices 96. Детали и рекомендации в issue `[lighthouse]`. Следующий прогон сравнивается с
> самой нижней строкой каждой таблицы.
