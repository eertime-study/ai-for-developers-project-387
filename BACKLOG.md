# Backlog

План дальнейшего развития «Календаря звонков». Базовый продукт (шаги 1–5) реализован;
здесь зафиксированы фичи и баги, которые на следующих шагах будут разбираться через
цикл **issue → triage → pull request → ревью** с участием OpenCode-агента.

Каждый пункт сформулирован так, чтобы его можно было один-в-один завести как GitHub issue.

> **Design First.** Пункты, помеченные ⚠️ **контракт**, меняют поведение API → сперва
> правится `spec/main.tsp` (источник правды), затем генерируются типы и реализуется
> фронт/бэк. Подробнее — в [CLAUDE.md](CLAUDE.md).

Приоритеты: **P1** — заметно влияет на пользователя/качество, брать первым; **P2** —
полезное расширение; **P3** — nice-to-have.

## 🐞 Баги

### BUG-1 · Молчаливое проглатывание ошибок бронирования · P1
`frontend/src/routes/BookingFormPage.tsx` (обработчик `createBooking.error`, ~строки 50–60)
показывает сообщение только для трёх кодов: `SLOT_ALREADY_BOOKED`, `INVALID_GUEST_EMAIL`,
`VALIDATION_ERROR`. Любая другая ошибка (сетевой сбой, 5xx, неожиданный код) **никак не
отображается** — кнопка просто разблокируется, и гость не понимает, забронировалось или нет.

**Ожидаемо:** общий fallback-алерт уровня формы («Не удалось забронировать, попробуйте ещё
раз») для всех необработанных ошибок, не ломая текущие частные случаи.
**Затронуто:** `frontend/src/routes/BookingFormPage.tsx`.

### BUG-2 · Подтверждение брони теряется при перезагрузке · P2 · ⚠️ контракт
`frontend/src/routes/BookingSuccessPage.tsx` берёт данные только из `location.state`. При
F5, прямом заходе по ссылке или возврате «назад-вперёд» state пуст → редирект на `/`, и
гость теряет детали брони (а письма-подтверждения сейчас тоже нет — см. FEAT-4).

**Ожидаемо:** подтверждение можно открыть по устойчивому адресу (например
`/bookings/success/:id`), подгрузив данные через `GET /bookings/{id}`.
**Затронуто:** `spec/main.tsp` (новая операция чтения брони), бэк, `BookingSuccessPage.tsx`,
роутинг.

## ✨ Фичи

### FEAT-1 · Отмена брони гостем · P1 · ⚠️ контракт
Сейчас бронь нельзя отменить: в контракте нет операции удаления, слот остаётся `booked`
навсегда. Нужна возможность гостю отменить запись, после чего слот снова становится
`available`.

**Ожидаемо:** `DELETE /bookings/{id}` (или отмена по токену из подтверждения) + экран/действие
отмены во фронте; отменённый слот освобождается.
**Затронуто:** `spec/main.tsp`, `backend/src/{bookings,store,service}.ts`, фронт.

### FEAT-2 · Редактирование и удаление типов событий владельцем · P2 · ⚠️ контракт
В админке есть только создание (`createEventType`). Владелец не может изменить заголовок/
описание/длительность или убрать неактуальный тип.

**Ожидаемо:** `PATCH /admin/event-types/{id}` и `DELETE /admin/event-types/{id}` + UI
редактирования/удаления в `AdminPage`. Продумать поведение для типов с существующими
бронями.
**Затронуто:** `spec/main.tsp`, `backend/src/{eventTypes,store,service}.ts`,
`frontend/src/routes/AdminPage.tsx`, `EventTypeCreateForm.tsx`.

### FEAT-3 · История прошедших встреч в админке · P3 · ⚠️ контракт
`listUpcomingBookings` отдаёт только предстоящие встречи. У владельца нет способа посмотреть
прошедшие.

**Ожидаемо:** просмотр прошедших встреч (отдельный запрос/параметр) + вкладка/фильтр в
админке.
**Затронуто:** `spec/main.tsp`, `backend/src/{bookings,service}.ts`,
`frontend/src/{routes/AdminPage,components/BookingsTable}.tsx`.

### FEAT-4 · Email-подтверждение гостю · P2
Экран успеха пишет «Мы получили ваши данные и добавили встречу в расписание»
(`BookingSuccessPage.tsx`), но письмо гостю не отправляется. Нужно реальное подтверждение
на email после успешной брони (и потенциально уведомление владельцу).

**Ожидаемо:** отправка письма после `createBooking` (провайдер/SMTP или dev-заглушка с
логом), без блокировки ответа API при сбое отправки.
**Затронуто:** `backend/src/{service,bookings}.ts` (+ модуль отправки почты).
